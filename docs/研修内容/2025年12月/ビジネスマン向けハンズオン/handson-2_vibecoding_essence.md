# Vibe Codingとは + 非エンジニアこそVibe Coder

## 話す内容

### 1. Vibe Codingの本質

**重要: Vibe Codingはプログラミングの話ではない**

色々なアウトプットを作るというだけだったら、別にプログラミングじゃなくてもいいじゃないかと思われるかもしれませんが、理由があります。

**なぜプログラミングなのか**:

まず、**プログラミング、要するにこの世の中は大体コードで制御されている**ので、コードに触れられるか触れられないかで、世界を動かす側に回れるか回れないかという壁があると思っています。これができるようになったら、あらゆることに応用ができる可能性が高いです、現代であれば。

もう1つが基礎力として、プログラミング学習をやる時に、どうしても**論理的に正しいものじゃないとプログラムは動かない**ので、この辺りをおのずと身につけられるというところが1つあります。

**コンテクストコントロールの習得**:

AIを使っていて「なんかいい出力が出ないな」「思ったのと違うんだよな」とChatGPTを使っていて思うことがあると思います。

本質的には、**コンテクスト（自分の思っている文脈）を正しい量、正しい質で渡せていない**というのが最も問題なんです。

例えば、「新規事業をやりたい」とAIに言った時に、何か出てきます。でもそこには、その人の思いも入っていなければ、会社の状況やそういう制約条件も入っていないので、結局何か違うとなります。

それをきちんと与えることで良い出力が出るわけなんですが、これを私は**コンテクストコントロール**と言っています。

プログラミングをAIにやらせるというのは、プログラムはそもそも**1行でも間違っていたらバグやエラーが出て動かない**ものです。だから地道に作り上げていく必要があるわけです。

つまり、**プログラムがアプリとして1本作れるレベルでAIから出力を引き出せる能力があれば、ドキュメントとして事業計画書を作る、企画書を作る、営業資料を作るというのはめちゃくちゃ簡単な話**なんです。

だからプログラミングを学んでおくことによって、実はそのスキルがそのまま色々な文系で我々が作るようなアウトプットを出すのにつながるということです。**どうせ触るんだったら一番難しいことやろうや**ということで、プログラムを作るということをみんなやろうよとお伝えしています。

### 2. 非エンジニアこそVibe Coderになるべき理由

#### 理由1: 解像度の飛躍的向上

**磨くべきは人間力とコンテクスト**:

AIで何か作りますと言うと、出てきたものを過小評価されることが結構ありますが、実は**磨かなきゃいけないのは経験値や人間力**です。

AIは大量のデータを学習してアウトプットを出す仕組みです。そういう意味では、どうしても**確率的に最も一般的な内容に収束**してくるんです。そこに違いを生み出すものは何かというと、やはり一人一人の**人生経験から出てくる思い**、**視点**、**目の前で起こっていることに対する課題意識**です。

これをインプットすることで、やっとAIの確率を自分の方向に修正する、引き寄せることができます。**編集者としての自分**を意識して使っていくことが重要です。

**Vibe Writingだけの場合**:
- 企画書は作れる
- 提案資料は作れる
- でも、「実際にどう作るのか」は他人任せ
- アイデアレベルの解像度に留まる

**Vibe Coding習得後**:
- **実際に使えるソフトウェアまで作れる**
- エンジニアの視点を持てる
- 「実装可能性」を含めて考えられる
- **実装レベルの解像度**に到達

| 能力 | Vibe Writing のみ | Vibe Coding 習得後 |
|------|-----------------|------------------|
| できること | 企画書・資料作成 | **ソフトウェア開発まで** |
| エンジニアとの会話 | 一方的な要求 | **対等な議論が可能** |
| プロトタイプ作成 | 外注待ち | **即座に自作可能** |
| 解像度 | アイデアレベル | **実装レベル** |

#### 理由2: コミュニケーションロスの消滅

**非エンジニアの壁**:

非エンジニアの方がアイデアを形にする壁はいくつかあります:

1. **人材確保の壁**: スタートアップを作る時、MVPを作って資金調達するにも、まずCTO的な人を見つけるという高いハードルがある
2. **コストの壁**: 安くたって数十万、普通だと数百万かかる
3. **コミュニケーションギャップ**: エンジニアにリソースを開けてもらって作ってもらっても、「自分が言ったのと違うよね」ということが起こる
4. **修正の壁**: ここにボタンをちょっとつけて欲しいだけなんだよというのが全然通らない。1度作ったものはなかなか修正が効かず、また何十万円取られる

**Vibe Coderになると**:
- プロトタイピング能力があれば、「**これを作って**」と見せられる
- 動くデモは言葉より強く、協力者を引き寄せる磁石になる
- **結果: 圧倒的に早い**

#### 理由3: 組織の「アイデアの墓場」問題の解決

**多くの組織で起きている現実**:

会議やSlackで「いいね」だけついて終わる、議事録には残っているけど誰も巻き取らないということがあると思います。世界的にVibe Codingでやっている人たちがなぜあんなに熱中しているのかというと、**眠らせてこざるを得なかったアイデアが、大体形的なものであれば形になっちゃう**というところです。これがいかに面白いかというところです。

**各部署に1人Vibe Coderがいると**:
- アイデアが出たその日に、プロトタイプを作れる
- 翌日の会議で「作ってみました」と見せられる
- 経営陣も具体的に判断できる
- **スピード感が激変**

**組織変革の実例**:
- 営業部門: 顧客管理ツールを自作、月末集計の残業ゼロ
- マーケティング部門: キャンペーン効果測定ダッシュボード、CPA 20%削減
- 事業企画部門: 競合分析ツール、週10時間の調査が15分に短縮

#### 理由4: ビジネス文書作成力も劇的に向上

**プログラミングで培った力がビジネス文書に転用される**:
- **完璧なコンテクスト構築力**: 企画書・提案書の質が別次元に
- **構造的思考**: 報告書の論理展開が明確に
- **精密な要件定義**: 曖昧さのない指示書・マニュアルが書ける

**実際の効果**:
- 企画書の通過率が上がる
- 提案書の説得力が増す
- 社内コミュニケーションの齟齬が減る

## 重要なメッセージ

> **「プログラムを書くことが目的ではない。AIから完璧な出力を得る技術を、最も難易度の高い環境で鍛えることで、あらゆるビジネスアウトプットの質が変わる」**

## 3. 今日使うツール: Antigravity（またはCursor）

### ツールについて

今日のハンズオンでは**Antigravity**を推奨していますが、**Cursorでも全く問題ありません**。

**Antigravity推奨の理由**:
- 無料で使える
- 今日の体験には十分な性能

**Cursorを使いたい方**:
- 有料プランでより良いモデルが選択できる
- 使い方はほぼ同じ
- **VS Code系のAIエディター（IDE）であれば本質は変わらない**

### なぜChatGPTではなくIDE型AIなのか

ここが今日最も重要なポイントです。

**ChatGPTとの決定的な違い**:

| 特徴 | ChatGPT（チャット型） | Antigravity/Cursor（IDE型） |
|------|---------------------|--------------------------|
| 出力形式 | テキストのみ | **複数ファイル自動生成** |
| コンテクスト管理 | 会話単位 | **プロジェクト全体** |
| 編集 | コピペして手動編集 | **その場で即座に修正** |
| プレビュー | 不可 | **ローカルで即座に確認** |
| 複雑な構造 | 難しい | **ディレクトリ構造も自動作成** |
| 継続性 | 会話が切れると消える | **ファイルとして永続保存** |

**結論**: **コンテクストコントロールが全て**

### コンテクストコントロールの本質

Vibe Codingで最も重要なのは、**コンテクストのコントロール**です。

**コンテクストコントロールとは**:
1. **どんなAIに何を投げるか**を徹底的に管理する
2. 情報を**蓄積・管理・生成**し続ける
3. 出てきたアウトプットから**新しいコンテクスト**を生み出す
4. それを次のAIへの情報として渡す

**この循環環境を作ることが、Vibe Codingの真髄です。**

#### ChatGPTの場合（コンテクストが断片的）

```
アイデア → ChatGPTに投稿 → テキスト出力 → コピペして保存
         ↓
      会話が終わる
         ↓
      コンテクスト消失
         ↓
    また1から説明...
```

#### IDE型AI（Antigravity/Cursor）の場合（コンテクストが蓄積・循環）

```
アイデア → プロジェクトフォルダ作成
         ↓
    企画書ファイル生成（コンテクストとして保存）
         ↓
    企画書を読み込んでLP生成（前のコンテクストを活用）
         ↓
    企画書＋LPを読み込んで説明資料生成（さらに蓄積）
         ↓
    すべてのファイルがプロジェクト内に保存
         ↓
    明日でも来週でも、このコンテクストから続きができる
```

**これが「コンテクストコントロール環境」です。**

### プログラミングで鍛える理由

プログラミングは**一文字でも間違えればエラー**です。

つまり、AIに対して：
- 曖昧な指示は通用しない
- 完璧なコンテクストを渡す必要がある
- ファイル間の整合性を保つ必要がある

**この厳しい環境でコンテクストコントロールを鍛えれば**、事業計画書も営業資料も「余裕」になります。

プログラミングで培った力：
- **情報の構造化能力**
- **曖昧さを排除する言語化力**
- **全体の整合性を保つ管理力**

これがすべてのビジネスアウトプットに転用されます。

### まとめ

**今日体験すること**:
- ✅ プロジェクトフォルダという「コンテクストの器」を作る
- ✅ ファイルが次々と生成され、コンテクストが蓄積される
- ✅ 蓄積されたコンテクストから新しいアウトプットが生まれる
- ✅ この循環が、ChatGPTでは不可能な「IDE型AIの真価」

**ツールはAntigravityでもCursorでもOK。大事なのは「コンテクストコントロール環境」を体験することです。**

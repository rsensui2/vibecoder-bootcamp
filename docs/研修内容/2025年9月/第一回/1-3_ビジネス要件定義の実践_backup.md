# **第一回**
## ビジネス要件定義の実践

---

## AI駆動開発の出発点：なぜ「要件定義」が重要なのか？

AIは優秀なアシスタントですが、**「何をすべきか」**を指示するのは人間の役割です。
そして、その指示書が**AI駆動開発における「要件定義」**です。

**要件定義 = 私たちの頭の中にあるアイデアやビジネス目標を、AIが正確に理解できる構造化されたテキスト（コンテクスト）に変換するプロセス**

この「指示書」の品質が、最終的なアウトプットの品質を決定します。

---

### なぜ「Markdown」で書くのか？

AIへの最高の指示書を作るための最適なフォーマットが**Markdown**です。

- **AIとの親和性:**
  `# 見出し`や`- リスト`で構造化されたMarkdownは、AIが文脈や階層構造を理解するのに最適です。
- **人間にとっての可読性:**
  Wordのように専門的なツールがなくても、誰でも読み書きできます。
- **バージョン管理:**
  Git（第二回で学習）で変更履歴を管理でき、チームでの共同作業にも向いています。

---

## AIとの対話でビジネス要件を洗練させる：5つのステップ

優れたビジネス要件は、最初から完璧である必要はありません。
**あなたのざっくりとしたアイデアをAIに投げかけ、対話を通じて洗練させる**
それがAI駆動開発における要件定義です。

---

### Step 1: ざっくりとしたアイデアをAIに話す

まずは、思いついたアイデアをそのままAIに伝えましょう。
完璧な文章である必要はありません。

**プロンプト例:**
```
自分が持っているガジェットを記録して、
他の人と共有できるアプリを作りたい。
買い物の記録にもなるし、
同じ趣味の人と繋がれたら面白そう。
```

---

### Step 2: AIに構造化してもらう

AIは、あなたの言葉を以下のような構造に整理してくれます。

**AIが整理する主な項目:**

#### プロジェクト概要
- **プロジェクト名**: アプリケーションの名前（仮でOK）
- **コンセプト**: このアプリを一言で表すと？
- **解決したい課題**: 誰の、どのような「不便」「不満」を解決するのか
- **ターゲットユーザー**: 最も喜んで使ってくれるのは誰か

#### 提供価値
- **コア機能**: 最も重要な機能（3つまで）
- **ユーザーにとっての価値**: ユーザーが得られるもの
- **ビジネスとしての価値**: どのように収益を上げるか

#### 制約条件
- **予算**: 想定している費用
- **期間**: いつまでにMVPが欲しいか
- **その他**: 技術的挑戦、デザインの好み、譲れないこと

---

### Step 3: フレームワークを使った深掘りを指示する

ざっくりとしたアイデアが構造化されたら、次はビジネスフレームワークを使ってAIに深掘りさせます。

**プロンプト例:**
```
7 Powersのフレームワークに基づいて、
私に質問しながらこのビジネス要件を深掘りしてください。
```

**AIからの質問例:**
- 「スケールメリット: ユーザーが増えるほど価値が高まる仕組みはありますか？」
- 「ネットワーク効果: 他のユーザーの存在が価値を生みますか？」
- 「ブランド: このアプリならではの独自性は何ですか？」
- 「スイッチングコスト: ユーザーが使い続ける理由は？」

---

### Step 4: AIに自分の知識でも提案させる

あなたが答えられない質問もあるはずです。その時は、AIにも考えさせましょう。

**プロンプト例:**
```
私がうまく答えられない部分は、
あなたの知識をもとに提案もしてください。
対話しながら一緒に練り上げましょう。
```

**AIの提案例:**
- 「ガジェット市場の分析によると、ユーザーは『所有の記録』より『他者との比較』に価値を感じる傾向があります」
- 「アフィリエイト収益モデルを組み込めば、ユーザーの購入行動から収益化できます」
- 「コレクション完成度を可視化する機能で、継続利用を促せます」

---

### Step 5: 対話で生まれた洞察をMarkdownで蓄積する

**最も重要なステップ**: AIとの対話で明らかになった情報は、**必ずMarkdownファイルに記録しましょう。**
これが、あなたとAIの**共通認識（コンテクスト）の資産**になります。

#### なぜMarkdownで記録するのか？
- 次回以降、このファイルをAIに読み込ませれば、同じ説明を繰り返さなくて済む
- Git管理できるので、要件の変化の履歴が残る
- チームメンバーとも共有できる
- プロジェクト全体像が常に可視化される

#### 記録すべき情報の例

**`business-requirements.md`**
```markdown
# プロジェクト概要
- **プロジェクト名**: GadgetLog
- **コンセプト**: ガジェット好きのための備忘録SNS
- **解決したい課題**: 所有ガジェットの管理と、同じ趣味の人との繋がり

# ターゲットユーザー
- デジタルガジェットを複数所有する30代〜50代の男性
- 新しいガジェット購入時に、過去の購入履歴を参照したい
- 他の人の機材セットを見て参考にしたい

# コア機能
1. ガジェット登録機能（写真、スペック、購入日、価格）
2. 他ユーザーとの共有・いいね機能
3. スペック比較機能

# 7 Powers分析（AIとの対話で明らかになった点）
- **ネットワーク効果**: ユーザーが増えるほど、参考になる機材情報が増える
- **データ資産**: ユーザーの購入履歴データが蓄積されると、レコメンド精度が上がる
- **ブランド**: 「ガジェット好き専用SNS」としてのニッチなポジション

# 収益モデル（AIの提案）
- アフィリエイトリンク（Amazon、楽天）
- プレミアムプラン（詳細な比較機能、過去の価格推移グラフ）

# 制約条件
- 予算: 月額$30以内（Supabase + Vercel）
- 期間: 1ヶ月でMVPリリース
- 技術: Next.js + Supabase、スマホ最適化必須
```

#### 次回からの使い方

次にAIと開発を再開するときは、このファイルをそのまま渡すだけ：

**プロンプト例（Cursor）:**
```
@business-requirements.md を読んで、
次に実装すべき機能について提案してください。
```

**このクセをつけることで:**
- AIとの対話が「使い捨て」ではなく「資産」になる
- プロジェクト全体像が常に可視化される
- チーム開発でも、全員が同じコンテクストを共有できる
- 開発の途中で方向性を見失わない

**これがVibe Coderの真髄です。**

```

各ドキュメントをMarkdown形式で構造化して出力してください。
```

```
📁 ゲートボール世界進出プロジェクト/
├── 📄 事業概要書.md
├── 📄 ビジネスモデル詳細.md
├── 📄 ターゲット顧客分析.md

└── 📄 リスク分析.md
```

**ポイント：** たった3分で、コンサルタントが2週間かける作業が完了！

### Step 3: 対話で要件を深める

#### ビジネスフレームワークを使った追加分析

```
「7 Powersの観点から競合優位性を分析して」

「Blue Ocean戦略で新市場を定義して」

「あなた自身にいくつか質問させてから、
さらに深い分析を提示して」
```

#### AIが自分に質問して深める例

**AIの自己質問：**
- なぜゲートボールは日本の高齢者に人気なのか？
- 海外のターゲットにとってのメリットは？
- 他のスポーツとの決定的な違いは？
- 文化的障壁をどう乗り越えるか？

**これらの質問を通じて、より深い洞察が生まれる！**

### 生成されるドキュメントの例

#### 事業概要書.mdの一部
```markdown
# ゲートボール世界進出プロジェクト

## ミッション
高齢者のスポーツとして日本で愛されるゲートボールを、

## 7 Powers分析
1. **ネットワーク効果**: プレイヤーが増えるほど楽しさが増す
```

```markdown
- **付け加える**: ARスコア表示、オンライン対戦
```

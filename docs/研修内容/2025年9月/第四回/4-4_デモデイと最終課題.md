# 4-4 デモデイと最終課題

## 概要
9月期の最終回は「ビルドとデプロイ」がテーマです。これまで積み上げてきた企画力・実装力・AI活用力を総動員し、安定したビルドと本番運用を見据えたデプロイまで完遂したうえで、デモデイで成果を共有します。本資料ではデモデイの目的、最終課題の要件、評価観点、そしてリリース準備のポイントを整理します。

## 学習目標
- デモデイの意義と発表で伝えるべきメッセージを理解する
- 最終課題の要件とアウトプットイメージを具体化する
- ビルド安定化からVercelデプロイまでを自走で完結できる
- 本番運用視点でのチェックリストや運用設計を身につける
- 学んだAI活用手法を業務・プロダクト開発へ横展開する

# 1. デモデイの目的：プロダクトを届ける最初の本番
単なる成果発表ではなく、デモデイを「初めての本番リリース」と位置づけましょう。

- **価値の証明**: アイデアを実装しビルドを通して本番 URL として届けることで、Vibe Coderとしての成果が証明されます。
- **信頼獲得**: 本番運用を想定した説明ができることで、チームメイトやステークホルダーの信頼を得られます。
- **学習の加速**: 他メンバーの発表を通じて、より洗練された開発・デプロイ手法を吸収できます。
- **次の一歩**: デプロイを完遂した経験が、今後の追加開発や社内展開の土台になります。

# 2. 最終課題：本番稼働を見据えたオリジナルWebアプリ開発

## 課題の目的
- **技術の統合**: フロントエンド・バックエンド・データベース・認証・デプロイの全工程をスルーで体験します。
- **再現性の獲得**: AI駆動開発のワークフロー（要件定義→設計→実装→レビュー→リリース）を自走で回し切ります。
- **運用準備**: ログ・メトリクスや権限設定など、公開後を意識した設計を取り入れます。

## 課題内容
テーマは自由です。第四回で扱う「実装レビュー」「Vercelデプロイ」「最新AI開発エコシステム」の内容を活かし、以下の観点を意識しましょう。

- ユーザーに提供したい価値は何か（ストーリーを言語化）
- 本番運用で壊れにくい構成か（例：環境変数、APIキー管理）
- 継続改善の余地をどこに残すか（ロードマップの提示）

### 必須要件
次の要素を最低限満たしてください。

1. **フロントエンド**: ユーザーが操作できるUI。Next.js + TypeScript推奨。
2. **バックエンド**: APIやサーバーサイドロジック。Next.js API RoutesまたはEdge Functions推奨。
3. **データベース**: 永続化層としてSupabase等を利用。
4. **認証**: Clerkなどを用いてサインアップ/ログインを提供。
5. **デプロイ**: Vercelへのデプロイまたは同等のホスティング環境で公開。

### ゴール設定
- **理想ゴール**: 本番用ドメイン（Vercel）で動作し、テストユーザーが利用できる状態。バックアップ手順やロールバック方法まで説明できる。
- **標準ゴール**: Vercel にデプロイ済みで、最低限のユーザーフローが通る。主要な環境変数や権限設定が整理されている。
- **最低限ゴール**: ローカルで動作し、ビルドコマンド `npm run build` が成功。デプロイ計画とタスク整理を提示する。

# 3. 開発プロセス：AIと走り切る5ステップ

## 1. 要件定義
ヒアリングテンプレートを活用し、ペルソナ・課題・価値をMarkdownで整理。AIには「ロール」「制約」「確認項目」を明示しましょう。

## 2. 設計
ER図、API仕様、UIフローをAIと共に作成。第四回の実装レビュー手法を用いて、レビュー観点（命名・責務・例外処理）を洗い出します。

## 3. 実装
小さな単位でプルリクを切る想定でコミット。AIへは「前提」「ゴール」「動作確認方法」を毎回渡して品質を確保します。

## 4. テスト・ビルド
`npm run lint` や `npm run test`、`npm run build` を実行し、CIを想定したチェックをローカルで再現。ビルドログはスクリーンショットやテキストで残しておくと発表で活用しやすくなります。

## 5. デプロイ
VercelでPreviewデプロイ→本番デプロイの順に実施。環境変数、ClerkのRedirect URL設定、SupabaseのRow Level Securityの確認をお忘れなく。

# 4. 発表形式：プロダクトとデプロイのストーリーを語る
- **時間**: 1人15分（発表10分 + 質疑応答4分 + 予備1分）
- **発表資材**: Markdown / HTMLベースのAI生成スライド（Cursor + Gensparkなど）。PowerPoint / Keynoteは禁止。
- **構成例**:
  1. プロダクト概要（課題・解決策・主要スクリーン）
  2. 技術構成（アーキテクチャ図、使用サービス）
  3. ビルド&デプロイで工夫した点（失敗談も歓迎）
  4. 今後のロードマップと導入アイデア
  5. デモ（実機操作 or 録画）

# 5. ビルドとデプロイの必須チェックリスト
- `npm run build` がローカルで通るか（CIで落ちる要因を排除）
- .env ファイルの本番用キーがVercelに設定されているか
- SupabaseのRLSポリシーやテーブルの初期データを用意したか
- Clerkダッシュボードで本番ドメインを許可したか
- リカバリ手順（例: 再デプロイ、以前のバージョンに戻す方法）をまとめたか
- 監視・ログ（Vercel Analytics、Supabase Logsなど）の確認方法を用意したか

# 6. 評価基準：価値・完成度・説明力
| 観点 | 質問例 | 期待するアウトプット |
|:---|:---|:---|
| ビジネス価値 | 誰のどんな課題をどのように解決したか | ストーリーとユーザーメリットが明確 |
| 完成度 | 安定して動作し、主要な導線が問題なく利用できるか | デモでつまずかず、Fallbackも説明できる |
| デプロイ品質 | ビルド・環境変数・依存サービスを適切に管理しているか | 本番URLとチェックリストを提示 |
| AI活用 | 開発プロセスでAIをどう活用したか | 指示テンプレートやプロンプトを共有 |
| プレゼン力 | 短時間で魅力と価値を伝えられるか | ストーリー性があり質疑に的確に回答 |

# 7. スケジュールとマイルストーン（例）
- **第4回講義当日**: 実装レビューで残タスク洗い出し・デプロイ計画策定
- **講義後3日以内**: Previewデプロイ、主要ユーザーフローのテスト完了
- **デモデイ5日前**: 本番デプロイ完了、バックアップ手順確立
- **デモデイ前日**: プレゼン資料確定、デモリハーサル実施

# 8. 提出物と締切
- GitHubリポジトリURL（READMEにセットアップ手順と使用技術を記載）
- デプロイしたアプリケーションのURL（理想・標準ゴールの場合）
- デモデイ用プレゼン資料（Markdown / HTML / Gensparkリンク）
- 本番運用チェックリスト（NotionやMarkdownで可）
- 締切: **デモデイ前日 23:59 JST**

# 9. 困ったときの対処法
- **ビルドエラー**: エラーログ全文をAIに渡し、原因特定→再発防止策まで相談。
- **デプロイ失敗**: Vercelの`Deployments`ログを確認し、環境変数・SSRエラー・権限漏れを順に潰す。
- **データベース不整合**: Supabaseの`Table Editor`でデータを確認し、MigrationファイルやSeedスクリプトを整備。
- **時間が足りない**: MVP優先度を再定義し、Must/Should/Wishで切り分ける。発表時は未対応項目も正直に共有。
- **質問したい**: Slackの`#vibecoder-qa`でスクリーンショットと状況を共有し、再現手順も併記する。

# 10. 最後に
ビルドとデプロイはゴールではなくスタートラインです。第四回で得たナレッジを活かし、本番を想定した仕上げとプロダクトストーリーでデモデイを成功させましょう。AIと共創しながら「自分にもここまでできた」という確信を手にし、次のプロジェクトへ踏み出してください。皆さんのリリース報告を楽しみにしています！

## 核心メッセージ

### このコラムで伝えるべき本質（1文）
**「要件定義=システム仕様書という誤解。本当に必要なのは『ビジネス要件』（誰のために、なぜ、何を解決するか）であり、システム要件（どう実装するか）はAIが生成できる」**

### 読後に読者に残すべき3つのポイント
1. **ビジネス要件 vs システム要件の区別**: 人間が考えるべきは前者、AIが助けてくれるのは後者
2. **非エンジニアが陥る罠**: 「どう実装するか」から考えてしまい、「なぜ作るか」が曖昧になる
3. **Context Controlの本質**: ビジネス要件を明確に言語化できれば、AIは最適なシステム要件を提案してくれる

## 構成指示

### タイトル案
1. **「要件定義の誤解を解く - システム仕様書を書く必要はない」**（推奨）
2. 「要件定義は難しくない - ビジネス要件だけ考えればいい」
3. 「非エンジニアが陥る『要件定義=技術仕様』の罠」

### 導入部（300-400字）
**狙い**: 読者の「要件定義って難しそう」という不安に共感し、その誤解を解く布石を打つ

**含めるべき要素**:
- 「要件定義」と聞いて、何を思い浮かべますか？
- 多くの非エンジニアは「システム仕様書」「ER図」「画面遷移図」などの技術文書を想像
- **しかし、それは誤解です**
- 非エンジニアが考えるべきは「ビジネス要件」だけ
- システム要件（技術的な詳細）は、AIが生成してくれる
- この誤解を解くことで、要件定義のハードルが一気に下がる

### 本論部1（500-600字）: ビジネス要件 vs システム要件

**狙い**: 2つの要件の違いを明確に定義し、読者の役割を明らかにする

#### 1. ビジネス要件とは（人間が考えるべき）
- **誰のために**: ターゲットユーザーは誰か？
- **なぜ**: どんな課題を解決するのか？どんな価値を提供するのか？
- **何を**: どんな機能があれば課題が解決するか？（ビジネスレベルの機能）
- **例**: 「営業担当者が顧客情報を素早く検索できるようにしたい」

**比喩**: レストランの企画
- ビジネス要件 = 「誰に（ビジネスパーソン）、なぜ（時短ランチ）、何を（500円で15分以内）提供するか」
- システム要件 = 「どう調理するか（キッチンレイアウト、調理器具、レシピ）」

#### 2. システム要件とは（AIが助けてくれる）
- **どう実装するか**: 技術スタック、データベース設計、API設計、UI/UX詳細
- **例**: 「PostgreSQLでusersテーブルを作り、全文検索インデックスを設定し、Next.jsのServer Actionsで検索APIを実装」
- **非エンジニアの誤解**: システム要件から考えてしまう
- **正しいアプローチ**: ビジネス要件を明確にすれば、AIが適切なシステム要件を提案してくれる

### 本論部2（400-500字）: 非エンジニアが陥る3つの罠

**狙い**: 具体的な失敗パターンを示し、読者が自分の行動を振り返るきっかけを作る

#### 罠1: 技術的な詳細から考えてしまう
- 「データベースはMySQLとPostgreSQLどっちがいい？」と悩む前に
- 「そもそも誰のために、何を解決するのか？」を明確にすべき
- **結果**: 技術選定に時間を取られ、肝心の価値提供が曖昧になる

#### 罠2: 「なぜ」が曖昧なまま進める
- 「とりあえずログイン機能」「とりあえず検索機能」
- **問題**: なぜそれが必要か？誰がどう使うか？が不明確
- **結果**: AIに指示しても、期待と違う出力（Garbage In, Garbage Out）

#### 罠3: 完璧な仕様書を書こうとする
- 「要件定義書は完璧でないとダメ」という強迫観念
- **真実**: ビジネス要件が明確なら、システム要件はAIと対話しながら詰められる
- **結果**: 完璧主義で一歩も進めない

### 実践パート（300-400字）: 今日からできる「ビジネス要件の明確化」

**狙い**: 読者がすぐに試せる実践的なアクションを提示

**実践ハック: 5W1Hでビジネス要件を整理**
```
1. Who（誰のために）: ターゲットユーザーは誰か？
2. Why（なぜ）: どんな課題を解決するのか？
3. What（何を）: どんな価値を提供するのか？
4. When（いつ）: どんなタイミングで使われるのか？
5. Where（どこで）: どんな環境で使われるのか？
6. How（どう）: どう使われるのか？（ビジネスレベルのフロー）
```

**例: 営業支援アプリ**
- Who: 中小企業の営業担当者（20-40代、IT不慣れ）
- Why: 顧客情報が散在し、商談前の準備に時間がかかる
- What: 顧客情報を一元管理し、素早く検索・閲覧できる
- When: 商談前、移動中、商談後のフォロー時
- Where: オフィス、外出先（モバイル対応必須）
- How: 顧客名や企業名で検索→詳細表示→メモ追加

**この5W1Hを明確にすれば、AIに「こういう課題を解決したい。どう実装すればいい？」と聞ける**

### 結論部（200-300字）: Context Controlとの繋がり

**狙い**: このコラムの内容をContext Controlの本質に結びつけ、次回への誘導を行う

**含めるべき要素**:
- ビジネス要件の明確化 = Context Controlの第一歩
- AIに正しいコンテクスト（背景、課題、目的）を与えることで、システム要件は自動的に導き出せる
- 「一文字でも間違えればエラー」というプログラミングの厳しさに対応するには、まず「何を作るか」が明確でなければならない
- 非エンジニアの強み: ビジネス課題を理解している。それをAIに正しく伝えればいい
- **次回予告**: 「では、ビジネス要件を明確にする具体的なプロセスは？5ステップで解説します」

## トーン＆スタイル指示

### 文体
- **会話的**: 「あなた」を主語に直接語りかける
- **挑発的だが丁寧**: 「誤解」を指摘するが、読者を否定しない（「誤解は当然。多くの人がそう思っている」）
- **共感から始める**: 「要件定義って難しそう...わかります」

### 使うべきレトリック
- **レストラン企画の比喩**: ビジネス要件（誰に、何を）とシステム要件（どう調理）の違い
- **対比**: ビジネス要件 vs システム要件、人間の役割 vs AIの役割
- **具体例**: 営業支援アプリ、顧客管理システムなど、ビジネスパーソンが想像しやすい例

### 避けるべき表現
- 「要件定義は簡単です」（不安を持つ読者を軽視する表現）
- 専門用語の羅列（ER図、正規化、制約など）を説明なしに使う
- 「AIに任せればOK」（人間の役割を軽視する表現）

### 必須要素
- **Garbage In, Garbage Out**: ビジネス要件が曖昧だと、システム要件も曖昧になる
- **Context Controlとの繋がり**: ビジネス要件を明確化することが、AIに正しいコンテクストを与えること
- **非エンジニアの強み**: ビジネス課題を理解している、それをAIに伝えればいい
- **180万円の価値**: 企業が180万円払うプログラムでは、この「ビジネス要件とシステム要件の分離」を徹底的に学ぶ

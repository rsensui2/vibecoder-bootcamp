**狙い**: バージョン管理の概念と、GitHubが生まれた背景を説明

#### バージョン管理とは何か

**問題**: 「最終版.docx」問題
- ファイルを上書きすると、過去のバージョンが消える
- 複数人で作業すると、誰が何を変更したかわからない
- 間違った変更をした場合、元に戻せない

**解決策**: バージョン管理システム
- **いつ**: 2024年11月19日 14:30
- **誰が**: 田中太郎
- **何を**: 「ログイン機能を追加」
- **どう**: `login.js`を新規作成、`app.js`を修正

**Git の革命**:
- Linus Torvalds（Linuxの開発者）が2005年に開発
- 「分散型バージョン管理システム」
- 各開発者がローカルに履歴を持ち、サーバーに依存しない

**GitHubの誕生**:
- 2008年、Git をクラウドで管理できるサービスとして誕生
- 「リモートリポジトリ」として、複数人で協働できる
- 2018年、Microsoftが75億ドルで買収
- 現在、1億人以上の開発者が利用

#### なぜGitHubが必要なのか？

**理由1: 履歴管理**
- 過去のバージョンにいつでも戻れる
- 「あの機能を追加する前の状態に戻したい」→ 簡単に戻せる

**理由2: 協働**
- 複数人で同じプロジェクトを編集できる
- 「田中さんが〇〇を変更、鈴木さんが△△を追加」→ 競合しない

**理由3: ドキュメント管理**
- コードだけでなく、ドキュメント、画像、設定ファイルも管理できる
- AIとのやり取り、要件定義書、設計図も全て履歴として残せる

### 本論部2（400-500字）: Vibe CoderにとってのGitHub

**狙い**: GitHubが「エンジニアだけのツール」ではなく、Vibe Coderにとっても必須であることを示す

#### Vibe CoderがGitHubを使う3つの理由

**理由1: AIとのやり取りの履歴を残す**
- Cursorでの開発プロセス全体を記録
- 「どのプロンプトで、どのコードが生成されたか」を追跡
- 失敗したプロンプトも記録し、改善の材料にする

**理由2: 要件定義・設計書の管理**
- ビジネス要件定義書、ER図、ワイヤーフレームをGitHubで管理
- 「いつ要件が変更されたか」を追跡
- 過去の要件定義に戻ることも可能

**理由3: いつでも過去の状態に戻れる**
- 「新機能を追加したら、既存機能が壊れた」→ 過去のバージョンに戻す
- Context Controlの4要素の1つ「いつでも引き出せるようにしておく」を実現

**Vibe Coderの成長プロセス**:
- Day 1: 要件定義をコミット
- Day 3: UI設計をコミット
- Day 7: 初期実装をコミット
- Day 14: 機能追加をコミット
- Day 21: リファクタリングをコミット

**この履歴があれば、いつでも「Day 7の状態」に戻れる**

### 実践パート（300-400字）: 今日からできる「GitHub活用」

**狙い**: 読者がすぐに試せる実践的なアクションを提示

**実践ハック: GitHubでプロジェクトを管理**

#### ステップ1: GitHubアカウント作成
- github.com でアカウント作成（無料）

#### ステップ2: リポジトリ作成
- 「New repository」をクリック
- プロジェクト名を入力（例: `learning-sns-app`）
- 「Public」または「Private」を選択

#### ステップ3: ローカルと連携
- Cursorでプロジェクトを開く
- ターミナルで`git init`、`git add .`、`git commit -m "Initial commit"`
- `git remote add origin [GitHubのURL]`
- `git push -u origin main`

#### ステップ4: コミットの習慣
- 機能を追加したら、`git commit -m "〇〇機能を追加"`
- 要件定義を更新したら、`git commit -m "要件定義を更新"`
- 毎日の終わりにコミット

**重要**: コミットメッセージは「何を変更したか」を明確に書く

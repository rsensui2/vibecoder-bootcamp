### 5.4 本論部2: 良いプロンプト vs 悪いプロンプト（800-900字）
**目的**: 具体例を示し、「良いプロンプト」の5つの要素を理解させる

**構成要素**:
- **悪いプロンプトの例**:
  ```
  「ボタンを作ってください」
  ```
  - 問題点:
    - どんなボタン? 色は? サイズは? クリックしたらどうなる?
    - AIは迷う → 期待と違う結果が出る
- **良いプロンプトの例**:
  ```
  「青い背景で、白いテキストの『送信』ボタンを作ってください。
  サイズは幅200px、高さ50pxです。
  クリックしたら、フォームデータをAPIに送信し、成功したら『送信しました』というアラートを表示してください。
  エラーが発生した場合は、エラーメッセージを赤字で表示してください。」
  ```
  - 良い点:
    - 具体的（色、サイズ、テキスト指定）
    - コンテクスト（フォームデータをAPIに送信）
    - 制約条件（成功時、エラー時の挙動）
    - 期待する出力形式（アラート、赤字表示）
- **良いプロンプトの5つの要素**:
  1. **具体性**: 曖昧な表現を避け、数値や固有名詞を使う
  2. **コンテクスト**: 「何のために」「どういう状況で」使うのかを説明
  3. **制約条件**: 「〜の場合は〜してください」という条件分岐を明示
  4. **期待する出力形式**: 「〜という形式で出力してください」を指定
  5. **例示**: 可能であれば、具体例を示す
- **さらに高度なプロンプト例**（Composer Modeでの指示）:
  ```
  「ToDo管理アプリに、優先度フィルター機能を追加してください。

  【要件】
  - 優先度は『高』『中』『低』の3段階
  - FilterBarコンポーネントを新規作成し、3つのボタンを配置
  - ボタンクリックで、該当する優先度のToDoのみ表示
  - 『すべて』ボタンも追加し、全ToDoを表示

  【技術要件】
  - types/todo.tsに`priority: 'high' | 'medium' | 'low'`を追加
  - components/FilterBar.tsxを新規作成
  - components/TodoList.tsxでフィルタリング処理を実装
  - lib/api.tsにフィルタリング用の関数を追加
  - pages/index.tsxでFilterBarを配置

  【デザイン要件】
  - 選択中のボタンは青い背景、未選択はグレー背景
  - ボタンのホバー時に色が薄くなる

  型定義の整合性を必ず保ってください。」
  ```
  - この例のポイント:
    - 要件、技術要件、デザイン要件を明確に分離
    - ファイル名、型定義まで具体的に指定
    - 「型定義の整合性を必ず保つ」という制約条件を明示

### 5.5 本論部3: プロンプトエンジニアリングの訓練方法（500-600字）
**目的**: 読者が「今すぐできる」訓練方法を示す

**構成要素**:
- **訓練ステップ**:
  1. **悪いプロンプトを意識的に改善する**:
     - 最初は曖昧な指示を出してみる
     - AIの結果を見て、「何が足りなかったか」を分析
     - 5つの要素を追加して、再度指示を出す
  2. **音声入力で訓練する**:
     - AquaVoiceで音声入力すると、「一気に話す」必要がある
     - → 頭の中で先に整理してから話す習慣がつく
     - → 言語化力が鍛えられる
  3. **Composer Modeで複雑なタスクに挑戦する**:
     - 複数ファイルにまたがる機能追加
     - 技術要件、デザイン要件を明確に分離
     - 型定義の整合性を意識
  4. **フィードバックループを回す**:
     - AIの結果を確認
     - 期待と違ったら、プロンプトを改善
     - 期待通りだったら、次はより複雑なタスクに挑戦
- **ポイント**:
  - 最初から完璧なプロンプトを書こうとしない
  - 試行錯誤を通じて、少しずつ改善していく
  - 「AIに怒らない」。期待通りに動かないのは、自分の言語化力が不足しているだけ

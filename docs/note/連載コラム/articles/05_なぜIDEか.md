# #5 なぜChatGPTでは不十分なのか - IDEという「プロジェクト全体のコンテクスト」

**前回**、Context Control Technology（言語化力 + AI使いこなし + ドキュメント管理）という総合力を理解し、「文脈を制する者が成果を制する」という原則を学びました。そして、プログラミング（★★★★★）で訓練すれば、ビジネス文書は「余裕」になることも。

でも、ここで疑問が湧きますよね。

**「ChatGPTでプログラミングのコードを書いてもらえるなら、それで十分じゃないの？」**

実際、ChatGPTに「Pythonで素数判定する関数を書いて」と頼めば、ちゃんとコードを返してくれます。では、なぜプロの開発者はChatGPTではなく、CursorのようなIDE型AIを使うのでしょうか？

答えは、**「プロジェクト全体のコンテクスト」という、チャット型AIでは扱えない次元のコンテクストが存在する**からです。

今回は、なぜIDEでContext Controlすることが必須なのかを理解しましょう。

---

## コンテクストには3つのレベルがある

Context Controlには、実は**3つのレベル**があります。

### Level 1: 単発指示（🔍 点）

**例**: ChatGPTに「Pythonで素数判定する関数を書いて」

- **コンテクストの範囲**: その1つの質問のみ
- **できること**: 単体の関数、単体のコード片
- **限界**: 前後関係がない、プロジェクトとの整合性がない

これは**点**のような文脈です。

---

### Level 2: 会話チェーン（🔗 線）

**例**: ChatGPTで連続会話
1. 「Pythonで素数判定する関数を書いて」
2. 「さっきの関数にエラーハンドリングを追加して」
3. 「さらにテストケースも書いて」

- **コンテクストの範囲**: 会話履歴（数ターン分）
- **できること**: 前のやり取りを踏まえた改善
- **限界**: 会話が長くなると前半を忘れる、プロジェクト全体は見えない

これは**線**のような文脈です。

---

### Level 3: プロジェクト全体（🌐 面）

**例**: Cursorがプロジェクトのすべてのファイルをインデックス化

- **コンテクストの範囲**: プロジェクト内の全ファイル、依存関係、設定ファイル
- **できること**: 複数ファイルの同時編集、プロジェクト全体の整合性維持
- **プログラミングの厳しさ**: 「一文字でも間違えればエラー」なので、Level 3のコンテクスト管理が必須

これは**面**のような文脈です。

---

**プログラミングでは、Level 3が必須です。**

なぜなら、プログラミングは「一文字でも間違えればエラー」だからです。

---

## ChatGPT vs Cursor：決定的な違い

では、具体的にどう違うのか？同じタスクで比較してみましょう。

### ChatGPT（Level 1-2）の場合

**タスク**: 「ログイン機能を作って」

**ワークフロー**:
1. ChatGPTに「ログイン機能を作って」と依頼
2. ChatGPTがコードを生成
3. **あなたがコピペして貼り付け**
4. エラーが出る
5. **エラーメッセージをChatGPTに貼り付け**
6. ChatGPTが修正案を提示
7. **また貼り付け...**（無限ループ）

**問題点**:
- プロジェクトの既存コードが見えない
- 依存関係が分からない
- ファイル間の整合性が取れない
- **Context Controlの訓練にならない**（断片的すぎる）

---

### Cursor（Level 3）の場合

**タスク**: 「ログイン機能を追加して」

**ワークフロー**:
1. Cursorに「ログイン機能を追加して」と依頼
2. **Cursorがプロジェクト全体を理解**
3. **関連ファイルを自動検出**
4. **複数ファイルを同時に編集**
5. **依存関係を自動解決**
6. **そのまま実行して動作確認**

**利点**:
- プロジェクト全体のコンテクストを保持
- 一貫性のあるコード生成
- **これこそが最高のContext Control訓練**（統合的な文脈管理）

---

この違い、わかりますか？

**ChatGPTは「質問→回答」のループ。Cursorは「プロジェクト全体を理解した上での実装」。**

---

## なぜプログラミングはLevel 3が必須なのか

### 「一文字でも間違えればエラー」の厳しさ

プログラミングは、容赦がありません。

- 変数名のスペルミス**1文字**でエラー
- インポート文の欠落**1行**でクラッシュ
- ファイル名の不一致で**ビルド失敗**

事業計画書なら、「こういう感じで」という曖昧さが許されます。

でも、プログラミングは違う。**一文字のミスも許されない。**

---

### プロジェクト全体の整合性が必須

さらに、プログラミングでは**ファイル間の整合性**が命です。

例えば、ファイルAで関数名を変更したら、ファイルB、C、Dでもその関数を呼び出している箇所をすべて修正しなければなりません。

**この連動をChatGPTは理解できません。**

なぜなら、ChatGPTはプロジェクト全体を見ていないからです。

**でも、CursorのようなIDEは自動的に全体を把握して整合性を保ちます。**

---

### Context Control訓練としての価値

前回学んだ難易度階層を思い出してください。

| タスク | 難易度 | Context Controlの厳しさ |
|--------|--------|------------------------|
| プログラム開発 | ★★★★★ | 一文字でも間違えればエラー |
| 事業計画書 | ★★★☆☆ | 曖昧でも「なんとなく」通じる |
| プレゼン資料 | ★★☆☆☆ | 前提が共有されていれば「察して」もらえる |
| メール文章 | ★☆☆☆☆ | 多少の不明確さは許容される |

**プログラミング（★★★★★）をIDEで訓練すれば、すべてが「余裕」になります。**

だからこそ、IDEという「Level 3のコンテクスト環境」で訓練する価値があるのです。

---

## 具体例で比較：ToDoアプリに「優先度フィルター機能」を追加

### ChatGPT（Level 1-2）の場合

```
あなた: 「優先度フィルター機能を追加して」
ChatGPT: 「このコードを追加してください：[コード]」

→ あなたがコピペ
→ エラー: useStateがインポートされていない
→ ChatGPTに「useStateのインポートエラーが出た」と報告
→ ChatGPTが修正案をくれる
→ また貼り付け...
→ エラー: 型定義が合わない
→ また報告...
```

**問題**: 断片的、手動コピペ、エラー対応が後手に回る

---

### Cursor（Level 3）の場合

```
あなた: 「優先度フィルター機能を追加して」

Cursor: プロジェクト全体をスキャン
  → 既存のフィルター機能を検出
  → useStateが既にインポート済みか確認
  → 型定義ファイルをチェック
  → 3つのファイルを同時に編集:
    - components/TodoList.tsx
    - hooks/useTodoFilter.ts
    - types/todo.ts

→ そのまま実行
→ 動作する完成品
```

**利点**: 統合的、自動整合性、一発で動く

---

この違いが、**「プロジェクト全体のコンテクスト」**の威力です。

---

## まとめ：Context Control訓練としてのIDE

**ChatGPT（Level 1-2）**: 単発質問や会話チェーンの文脈。断片的。

**Cursor（Level 3）**: プロジェクト全体の文脈。統合的。

---

**Context Control訓練としての価値**:

IDEでプロジェクト全体を統合的に管理する能力。

これが**最高難度のContext Control訓練**です。

プログラミング（★★★★★）をIDEでマスターすれば、事業計画書（★★★☆☆）は「余裕」になります。

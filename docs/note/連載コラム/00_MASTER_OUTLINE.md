## 📋 プロジェクト概要

### 目的
大学版Vibe Coding講義（180万円/人企業向けプログラムと同等）の核心哲学を基に、企業経営者・ビジネスパーソンに向けて「Context Control Technology」としてのVibe Codingの本質を伝えるNote連載コラムシリーズ。

**最終ゴール**: [vibecoderbootcamp.com](https://vibecoderbootcamp.com) への誘導、Bootcamp受講を促進

### ターゲット読者
- 企業経営者、事業責任者
- ビジネスパーソン（非エンジニア）
- AI駆動開発に興味があるが「プログラミング学習が必要」という先入観を持つ層
- アイデアはあるが形にできない「他力本願の苦しみ」を経験している人
- 10年間スタートアップ経営で「アイデア→形成化のステップが遠すぎる」問題に直面してきた人

### コンテンツ戦略
- 毎日1本ずつ公開（読者の知的好奇心を持続させる）
- 1本1,500-2,500字（通勤・休憩時間で読める）
- 各コラムは独立しつつ、全体で一貫したストーリーを形成
- 大学版オリエンテーション（180万円/人プログラム）のエッセンスを一般向けに再構成

## 🎯 核心メッセージ（全コラム共通）

### このシリーズで伝える本質（1文）
**「Vibe Codingのゴールはプログラミングではなく、Context Control Technology（コンテクストコントロール）という総合力の習得である。プログラミングは『一文字でも間違えればエラー』という厳しさゆえに、最高の訓練題材に過ぎない」**

### 読者に残すべき3つの認識転換

1. **プログラミングは手段、Context Controlが目的**
   - プログラムが作れること自体はゴールではない
   - AIから高品質な出力を引き出す「総合力」が本質
   - 学ぶべきは「言語化力 + AI使いこなし + ドキュメント管理」

2. **非エンジニアの構造的ハンディキャップからの解放**
   - 「アイデア→形成化のステップが遠すぎる」問題
   - 他力本願（エンジニア依存・外注依存・資金依存）→自律創造へのパラダイムシフト
   - 泉水様の10年間の苦しみ：「毎日アイデアが生まれるのに、形にできない」

3. **ビジネス文書は「余裕」になる**
   - 最難関（プログラミング）をマスターすれば、事業計画書・提案書は容易
   - プログラム（★★★★★）→ 事業計画書（★★★☆☆）→ プレゼン資料（★★☆☆☆）→ メール文章（★☆☆☆☆）
   - Context Controlは業務全般に応用可能

## 【セクション2】IDE×Context Control実践 - 具体的手法の提示（#5-10）

**セクションゴール**: IDE型AIという具体的ツールを通じて、Context Controlを実践する方法を示す

### #5 なぜIDEでコンテクストコントロールするのか
- **文字数**: 2,000字
- **核心メッセージ**: チャット型AI（ChatGPT）では限界がある。プロジェクト全体の文脈を扱うには、ファイル構造やコード全体を見渡せるIDE型AIが必須
- **出典**: 大学版オリエンテーション（IDE型AIとは何か）
- **チャット型 vs IDE型の決定的な違い**:
  - 理解範囲：1つの会話 vs プロジェクト全体
  - コード反映：コピペ必要 vs 自動反映
  - 用途：学習・アイデア出し vs 本格開発
- **重要な認識**: チャット型AIだけでは「勉強」はできても「開発」はできない
- **具体例**: Cursorでの実際の開発フロー
- **次回への誘導**: 「IDEの中でも、Composer Modeという機能が革命的です」

### #6 Composer Mode - プロジェクト全体を見渡すAI
- **文字数**: 1,800字
- **核心メッセージ**: Composer Modeはプロジェクトのすべてのファイルをコンテクストとして理解し、一貫性のある実装を実現する
- **出典**: ライト版・大学版のCursor実装部分
- **Composer Modeの3つの強み**:
  1. 複数ファイルの同時編集
  2. プロジェクト全体の一貫性維持
  3. 自動的なコンテクスト管理
- **実践ハック**: Composer Modeでの指示の出し方
- **次回への誘導**: 「では、そのComposerに『何を』伝えればいいのでしょうか？」

## 【セクション3】仕組みを理解する - 深層理解で知的好奇心を満たす（#11-16）

**セクションゴール**: 表層的な使い方だけでなく、LLM・要件定義・Webアプリの仕組みを理解することで、応用力と知的満足を得る

### #11 要件定義の誤解を解く
- **文字数**: 1,800字
- **核心メッセージ**: 「要件定義=システム仕様書」という誤解。本当に必要なのは「ビジネス要件」であり、システム要件はAIが生成できる
- **出典**: 大学版・ライト版ビジネス要件定義
- **ビジネス要件 vs システム要件**:
  - ビジネス要件：人間が考えるべき（誰のために、なぜ、何を解決するか）
  - システム要件：AIが生成できる（どう実装するか）
- **非エンジニアが陥る罠**: システム要件から考えてしまう
- **次回への誘導**: 「では、ビジネス要件をどう定義するか？5ステップで解説します」

### #17 データベースという永続化の仕組み
- **文字数**: 2,000字
- **核心メッセージ**: データベースは「データを永続的に保存する」仕組み。RDB（リレーショナルDB）の概念を理解すれば、AIへの指示が明確になる
- **出典**: 大学版・ライト版データベース基礎

  - リレーション = 貸出記録と本の紐付け
- **なぜデータベースが難しいか**: 正規化、リレーション、制約 → AIが助けてくれる
- **次回への誘導**: 「では、実際にSupabaseというツールでDBを使ってみましょう」

## 🎨 トーン＆スタイルガイド

### 全体的なトーン
- **挑発的だが、丁寧**: 既成概念を覆すが、読者を否定しない
- **具体的で実践的**: 抽象論だけでなく、必ず実践ハックを含む
- **知的好奇心を刺激**: 表層→構造→本質の3層で掘り下げる
- **泉水様の人格を反映**:
  - 非エンジニア出身
  - 10年の他力本願の苦しみ（「毎日アイデアが生まれるのに、形にできない」）
  - 2022年12月まで未経験、2年で80本のアプリ
  - 「2022年12月まで、コードを1行も書けませんでした。そして、今も書けません。」

### 文体の特徴
- 会話的（「あなた」を主語に直接語りかける）
- 短文と長文を織り交ぜてリズムを作る
- 命令形を効果的に使う（「～するな」「～しよう」）
- 比喩を多用（レストラン、水槽、オーケストラ、建築、図書館、ホテル、出版等）

### 避けるべき表現
- 専門用語の羅列（必ず日常的な比喩で説明）
- 「簡単」「誰でもできる」といった安易な楽観論
- プログラミング学習そのものを否定する表現（「あなたは学ばなくていい」というスタンス）
- AIへの過度な依存を推奨する表現（「AIを利用しながら学び、いずれはAIを凌駕する」）

### 必須要素
- 各コラムに「実践ハック」を含める（すぐ試せるアクション）
- 次のコラムへの自然な誘導文
- 具体例・メタファーを必ず含める
- 「なぜ」を3層で掘り下げる（表層→構造→本質）
- **大学版の核心哲学を反映**:
  - 本質はContext Control、プログラミングは訓練題材
  - 一文字でも間違えればエラー = 最高の訓練

  - 他力本願からの脱却

# プロンプト評価テンプレート

このテンプレートは、プロンプトを5つの軸で評価するための詳細な基準を提供します。

---

## 評価軸と配点

各軸を0-10点で評価し、合計50点満点で採点します。

---

## 1. 明確性（Clarity） - 10点満点

### 評価基準

| スコア | 評価 | 説明 |
|--------|------|------|
| 9-10 | 優秀 | 曖昧さが全くなく、誤解の余地がない |
| 7-8 | 良好 | 概ね明確だが、1-2箇所の曖昧さ |
| 5-6 | 普通 | 複数の曖昧な表現があり、解釈の余地 |
| 3-4 | 不十分 | 多くの曖昧な表現、誤解を招く可能性 |
| 0-2 | 不可 | ほとんど理解不能 |

### チェックポイント

```yaml
clarity_checklist:
  ✅ すべての指示が具体的か？
  ✅ 抽象的な表現を避けているか？
  ✅ 専門用語が定義されているか？
  ✅ 「適切に」「良く」など曖昧な修飾語を避けているか？
  ✅ 数値や基準が明示されているか？
```

### 採点例

#### 10点の例
```markdown
## コードレビュー指示

以下の3つの観点でコードをレビューしてください：

1. **セキュリティ**: XSS、SQLインジェクション、CSRF脆弱性
2. **パフォーマンス**: O(n²)以上のアルゴリズム、不要なループ
3. **可読性**: 変数名の明確さ（最低3文字）、コメント比率20%以上
```

#### 5点の例
```markdown
## コードレビュー指示

コードを適切にレビューしてください。
問題があれば指摘してください。
```

## 2. 完全性（Completeness） - 10点満点

### 評価基準

| スコア | 評価 | 説明 |
|--------|------|------|
| 9-10 | 優秀 | すべての要素が網羅され、例外ケースも考慮 |
| 7-8 | 良好 | 主要な要素は網羅、一部の例外ケースが不足 |
| 5-6 | 普通 | 基本要素のみ、詳細や例外が不足 |
| 3-4 | 不十分 | 重要な要素が複数欠落 |
| 0-2 | 不可 | ほとんどの要素が欠落 |

### チェックポイント

```yaml
completeness_checklist:
  ✅ すべての必須要素が含まれているか？
  ✅ 例外ケースが考慮されているか？
  ✅ 制約条件が明記されているか？
  ✅ 入力・出力の形式が指定されているか？
  ✅ 品質基準が定義されているか？
  ✅ エラー処理の指示があるか？
```

### 採点例

#### 10点の例
```markdown
## 講義資料生成

### 必須要素
1. タイトルとメタ情報（講義番号、日付、所要時間）
2. 学習目標（3-5個、「〜できる」形式）
3. 前提知識（必須と推奨を分ける）
4. 本文（3セクション以上）
5. まとめ（重要ポイント3つ、次のステップ）

### 品質基準
- 各セクションに1つ以上の具体例
- 2つ以上のMermaid図
- 1つ以上の比較表

### 制約
- 全体: 3000-5000文字
- 1文: 30文字以内
- 段落: 150文字以内

### 例外処理
- 初心者向け: 用語解説を追加
- 上級者向け: 応用例を追加
```

#### 5点の例
```markdown
## 講義資料生成

タイトル、内容、まとめを含めてください。
```

## 3. 構造性（Structure） - 10点満点

### 評価基準

| スコア | 評価 | 説明 |
|--------|------|------|
| 9-10 | 優秀 | 論理的で段階的、読みやすい構造 |
| 7-8 | 良好 | 概ね構造化されているが、一部に改善余地 |
| 5-6 | 普通 | 基本的な構造はあるが、論理の飛躍あり |
| 3-4 | 不十分 | 構造が不明瞭、論理的繋がりが弱い |
| 0-2 | 不可 | 構造がない、ランダムな指示の羅列 |

### チェックポイント

```yaml
structure_checklist:
  ✅ セクション分けが明確か？
  ✅ 段階的な指示になっているか？
  ✅ 論理的な流れがあるか？
  ✅ 各セクションの目的が明確か？
  ✅ セクション間の繋がりが自然か？
  ✅ 視覚的に読みやすいか（見出し、箇条書き）？
```

### 採点例

#### 10点の例
```markdown
## コンテンツ生成プロセス

### フェーズ1: 分析（5分）
既存資料を読み込み、以下を分析：
- 対象読者
- 学習目標
- トピックの難易度

### フェーズ2: 設計（10分）
分析結果に基づいて設計：
- 章立て構成（3-5章）
- 各章の学習目標
- 使用する図表の種類

### フェーズ3: 生成（20分）
設計に従って生成：
- 章ごとに順番に作成
- 各章終了時に品質チェック
- 不足要素があれば追加

### フェーズ4: 検証（5分）
最終確認：
- チェックリストで品質確認
- 例と図が十分か確認
- 誤字脱字チェック
```

#### 5点の例
```markdown
## コンテンツ生成

既存資料を読んでください。
それから新しいコンテンツを作ってください。
最後に確認してください。
```

## 4. 例示性（Examples） - 10点満点

### 評価基準

| スコア | 評価 | 説明 |
|--------|------|------|
| 9-10 | 優秀 | 豊富で具体的な例、良い例と悪い例の両方 |
| 7-8 | 良好 | 複数の例があるが、一部が抽象的 |
| 5-6 | 普通 | 最低限の例のみ、悪い例がない |
| 3-4 | 不十分 | 例が1-2個のみ、不適切な例 |
| 0-2 | 不可 | 例がない、または全く参考にならない |

### チェックポイント

```yaml
examples_checklist:
  ✅ 良い例が3つ以上あるか？
  ✅ 悪い例が2つ以上あるか？
  ✅ 実践的な例が1つ以上あるか？
  ✅ 各例に説明が付いているか？
  ✅ 例が多様か（異なるシナリオ）？
  ✅ コード例は動作するか？
```

### 採点例

#### 10点の例
```markdown
## コミットメッセージの書き方

### ✅ 良い例

#### 例1: 機能追加
\`\`\`
feat: ユーザー認証にGoogle OAuthを追加

- Clerk SDKを統合
- ログインボタンをヘッダーに配置
- 認証状態の永続化を実装
\`\`\`
**なぜ良い**: プレフィックス明確、概要簡潔、詳細は箇条書き

#### 例2: バグ修正
\`\`\`
fix: 投稿削除時のメモリリークを修正

リスナーの解除漏れが原因。useEffect のクリーンアップで対処。
\`\`\`
**なぜ良い**: 問題と解決策が明確

#### 例3: リファクタリング
\`\`\`
refactor: API呼び出しをカスタムフックに抽出

useApiCall フックを作成し、エラーハンドリングとローディング状態を統一。
\`\`\`
**なぜ良い**: 目的と手段が明確

### ❌ 悪い例

#### 例1: 不明瞭
\`\`\`
update
\`\`\`
**なぜダメ**: 何を更新したか不明

#### 例2: 詳細すぎ
\`\`\`
ログインボタンの色を#3B82F6から#2563EBに変更し、
ホバー時の効果を追加し、フォントサイズを14pxから16pxに...
\`\`\`
**なぜダメ**: 実装詳細すぎ、概要が不明

### 💡 実践例
\`\`\`bash
# 機能開発の完全なコミット例
git add src/auth
git commit -m "feat: Google OAuth認証を実装

- Clerk SDKを統合（v4.2.0）
- /login エンドポイント追加
- 認証状態をReduxで管理
- テストカバレッジ85%達成

Closes #123"
\`\`\`
```

#### 5点の例
```markdown
## コミットメッセージの書き方

良い例:
\`\`\`
feat: 新機能追加
\`\`\`

悪い例:
\`\`\`
update
\`\`\`
```

## 5. 実行可能性（Actionability） - 10点満点

### 評価基準

| スコア | 評価 | 説明 |
|--------|------|------|
| 9-10 | 優秀 | すぐに実行可能、具体的なアクション指向 |
| 7-8 | 良好 | 実行可能だが、一部に曖昧さ |
| 5-6 | 普通 | 実行可能だが、解釈が必要 |
| 3-4 | 不十分 | 抽象的で、実行が困難 |
| 0-2 | 不可 | 実行不可能、何をすべきか不明 |

### チェックポイント

```yaml
actionability_checklist:
  ✅ 各指示が動詞で始まっているか？
  ✅ 成果物が明確か？
  ✅ 判断基準が具体的か？
  ✅ ステップが実行可能か？
  ✅ ツールや手法が指定されているか？
  ✅ 成功の定義が明確か？
```

### 採点例

#### 10点の例
```markdown
## APIエンドポイントの実装

### ステップ1: スキーマ定義（10分）
\`src/schema/user.ts\` を作成し、以下を定義：
\`\`\`typescript
interface User {
  id: string;
  email: string;
  createdAt: Date;
}
\`\`\`

### ステップ2: エンドポイント作成（20分）
\`src/api/users.ts\` を作成し、以下を実装：
- GET /api/users - 全ユーザー取得
- POST /api/users - ユーザー作成
- エラーハンドリング（400, 500）

### ステップ3: テスト作成（15分）
\`src/api/__tests__/users.test.ts\` を作成：
- 正常系テスト: 3ケース
- 異常系テスト: 2ケース
- カバレッジ: 80%以上

### 成功の定義
- [ ] すべてのテストが合格
- [ ] TypeScript エラーなし
- [ ] ESLint 警告なし
```

#### 5点の例
```markdown
## APIエンドポイントの実装

APIを実装してください。
ユーザー情報を扱えるようにしてください。
テストも書いてください。
```

## 総合評価

### スコア計算

```python
total_score = (
    clarity_score +
    completeness_score +
    structure_score +
    examples_score +
    actionability_score
)
```

### 総合判定

| 総合スコア | 判定 | 推奨アクション |
|-----------|------|---------------|
| 45-50 | ⭐⭐⭐⭐⭐ 優秀 | そのまま使用可 |
| 40-44 | ⭐⭐⭐⭐ 良好 | 微調整推奨 |
| 35-39 | ⭐⭐⭐ 普通 | 改善必要 |
| 30-34 | ⭐⭐ 不十分 | 大幅改善必要 |
| 0-29 | ⭐ 不可 | 再作成推奨 |
